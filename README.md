# proxmox-demov2

Terraform configuration for a Proxmox home lab. Provisions a jump host and one or more isolated target VMs, with snapshot-based reset to a clean baseline.

## Architecture

```
Your machine
    │
    │ SSH (<jump-host-ip>)
    ▼
lab-ubuntu  ── vmbr0 (LAN, DHCP) ──► internet
    │
    │ vmbr1 (isolated, 10.10.10.1/24)
    ▼
target-01 (10.10.10.11)   ← no internet, reachable only through lab-ubuntu
target-02 (10.10.10.12)   ← add more by editing locals.target_vms
...
```

- **`lab-ubuntu`** — dual-homed jump host. Has a public LAN IP via DHCP on `vmbr0` and a static `10.10.10.1/24` on the isolated `vmbr1` bridge.
- **`vmbr1`** — an isolated Linux bridge with no physical uplink. Target VMs have no internet access.
- **`ubuntu-jammy-template`** — a Proxmox VM template built from the Ubuntu 22.04 cloud image. Never boots directly; target VMs are full clones of it.
- **Target VMs** — cloned from the template, isolated on `vmbr1`, reachable via SSH through lab-ubuntu as a ProxyJump. Each gets a static IP set by cloud-init. A `clean` Proxmox snapshot is taken automatically after first boot so the VM can be reset at any time.

## Prerequisites

- Proxmox VE host (tested on 9.1.11) with:
  - Two datastores: `local` (dir, for ISOs and snippets) and `local-zfs` (ZFS, for VM disks)
  - A PAM user (e.g. `terraform`) with an API token and SSH key access
  - A sudoers rule at `/etc/sudoers.d/<user>-terraform` granting passwordless access to: `tee /var/lib/vz/*`, `tee /etc/pve/*`, `pvesm`, `qm`, `pvesh`, `ifreload`, `ifup`, `ifdown`
- Terraform >= 1.3.0
- SSH agent running with your key loaded (`ssh-add`)

## Setup

**1. Copy and fill in variables:**

```bash
cp terraform.tfvars.example terraform.tfvars   # if you have an example
# or create terraform.tfvars directly:
```

```hcl
pve_endpoint    = "https://<pve-host-ip>:8006/"
pve_api_token   = "<user>@pam!TOKEN_NAME=TOKEN_SECRET"
ubuntu_password = "yourpassword"
```

The API token format is `username@realm!token_name=secret_uuid`. Find your token name with:

```bash
ssh <pve-user>@<pve-host-ip> 'pveum user token list <user>@pam'
```

**2. Initialize and apply:**

```bash
terraform init
terraform apply
```

First apply takes several minutes — it downloads the Ubuntu cloud image to the Proxmox host, creates all VMs, waits for cloud-init to finish, and takes the baseline snapshot.

## SSH access

**To the jump host:**
```bash
ssh ubuntu@<jump-host-ip>
```

**To a target VM (through the jump host):**
```bash
ssh -J ubuntu@<jump-host-ip> ubuntu@10.10.10.11
```

**Recommended `~/.ssh/config` entry:**
```
Host 10.10.10.*
  User ubuntu
  ProxyJump ubuntu@<jump-host-ip>
  StrictHostKeyChecking no
  UserKnownHostsFile /dev/null
```

With that in place: `ssh ubuntu@10.10.10.11` works directly from your machine.

## Adding more target VMs

Edit the `locals` block in `main.tf`:

```hcl
locals {
  target_vms = {
    "target-01" = "10.10.10.11"
    "target-02" = "10.10.10.12"   # add a line
    "target-03" = "10.10.10.13"
  }
}
```

Then `terraform apply`. Terraform will clone the template, upload a cloud-init snippet, boot the VM, wait for cloud-init, take a snapshot, and regenerate `reset.sh` — all automatically.

## Resetting target VMs

After `terraform apply`, a `reset.sh` script is generated in this directory with the Proxmox VM IDs baked in.

```bash
./reset.sh              # reset all targets to the clean snapshot
./reset.sh target-01    # reset one specific target
```

Reset stops the VM, rolls back the disk and config to the `clean` snapshot, and restarts it. Takes about 10 seconds per VM.

## File structure

```
versions.tf          — provider version pins (bpg/proxmox, null, local)
providers.tf         — provider config (endpoint, API token, SSH agent)
variables.tf         — variable declarations
main.tf              — all resources (jump host, template, targets, snapshots, reset script)
outputs.tf           — vm_ipv4_address (lab-ubuntu LAN IP), target_vm_ips (static map)
terraform.tfvars     — secret values (gitignored)
reset.sh             — generated by terraform apply, not hand-edited
```

## How it works internally

### Cloud image download

`proxmox_download_file` fetches the Ubuntu 22.04 cloud image directly onto the Proxmox host into the `local` datastore. `overwrite = false` means subsequent applies skip the download if the file already exists.

### Jump host (lab-ubuntu)

A standard VM built directly from the cloud image (no template). Has two NICs:
- `vmbr0` — gets a LAN IP via DHCP. The QEMU guest agent reports this IP back to Terraform, which surfaces it as the `vm_ipv4_address` output.
- `vmbr1` — configured with a static `10.10.10.1/24` after first boot via a `null_resource` that SSHes in and writes a netplan config.

Cloud-init on lab-ubuntu installs `qemu-guest-agent`, `net-tools`, and `curl` from the internet on first boot.

### Isolated bridge (vmbr1)

`proxmox_network_linux_bridge` creates a Linux bridge on the Proxmox host with no physical port attached. VMs on this bridge can talk to each other but have no path to the LAN or internet.

### Template

`ubuntu-jammy-template` is created from the same cloud image with `template = true` and `started = false`. It exists solely as a clone source — Proxmox marks it as a template, which prevents it from being started directly and enables fast full clones.

Note: the template has no packages pre-installed (it never boots to run cloud-init). Target VMs therefore also have no extra packages. The QEMU guest agent is not installed on target VMs, so `agent { enabled = false }` is set to prevent Terraform from hanging waiting for agent network data.

### Target VMs

Each entry in `locals.target_vms` produces:
1. A cloud-init snippet uploaded to the Proxmox `local` datastore — sets hostname, ubuntu user, SSH key, and password.
2. A full clone of the template with the snippet attached and a static IP configured via the `initialization` block.

The static IP (`10.10.10.x/24`) is set by cloud-init metadata, not DHCP. No gateway is configured — there is no routing out of the isolated network by design.

### Baseline snapshots

After each target VM boots, a `null_resource` SSHes to the VM through lab-ubuntu as a bastion and runs `cloud-init status --wait`. Once cloud-init reports done, a `local-exec` provisioner SSHes to the Proxmox host and runs `qm snapshot <vmid> clean`. This snapshot captures the full disk state at the known-good post-cloud-init point.

The snapshot resource is triggered by VM ID, so it re-runs (and creates a new `clean` snapshot) only if the VM is destroyed and recreated.

### Reset script

`local_file.reset_script` generates `reset.sh` using Terraform's template engine to inject the current VM IDs. Running `./reset.sh` calls `qm rollback <vmid> clean --start` on the Proxmox host for each target, which stops the VM, restores disk and config from the snapshot, and restarts it.

## Known limitations

- Target VMs have no internet access and no package manager connectivity. To install packages, you must either transfer files through lab-ubuntu or rebuild the template with packages pre-installed (boot the template VM once on a LAN-connected bridge, run apt, shut down, re-mark as template).
- The `null_resource` for lab-ubuntu's vmbr1 IP uses a known interface name (`enp6s19`). If your Proxmox host assigns a different name, you'll need to update the netplan inline in `main.tf` and re-taint the resource.
- `terraform.tfvars` is gitignored. Do not commit credentials.
